# Issues (Bug Reports & Feature Requests)

This folder tracks bugs, feature requests, and technical debt.

## Purpose

Issues document **PROBLEMS** and **IMPROVEMENTS**. They ensure nothing falls through the cracks.

## When to Create an Issue

- ✅ Found a bug
- ✅ User reported a problem
- ✅ Feature request from stakeholder
- ✅ Technical debt identified

## Issue Templates

### Bug Report

```markdown
# [BUG-NNN] Brief Description

**Type:** Bug
**Priority:** High/Medium/Low
**Status:** Open/In Progress/Resolved
**Reporter:** Name
**Assignee:** Name
**Date Reported:** YYYY-MM-DD

## Description
Clear description of the bug.

## Steps to Reproduce
1. Step 1
2. Step 2
3. Step 3

## Expected Behavior
What should happen?

## Actual Behavior
What actually happens?

## Environment
- **OS:** Windows/Mac/Linux
- **Browser:** Chrome/Firefox/Safari
- **Version:** X.Y.Z

## Screenshots
[Add screenshots if helpful]

## Logs/Errors
```
Error messages or logs here
```

## Root Cause
[To be filled during investigation]

## Solution
[To be filled when fixed]

## Related Issues
- Related to #123
- Blocks #456
```

### Feature Request

```markdown
# [FEATURE-NNN] Brief Description

**Type:** Feature Request
**Priority:** High/Medium/Low
**Status:** Proposed/Accepted/In Progress/Completed
**Requester:** Name
**Date Requested:** YYYY-MM-DD

## Problem Statement
What problem does this solve?

## Proposed Solution
How could we solve it?

## Alternative Solutions
1. Alternative 1
2. Alternative 2

## Benefits
- Benefit 1
- Benefit 2

## Drawbacks
- Drawback 1
- Drawback 2

## User Stories
- As a [user type], I want [goal] so that [benefit]

## Success Criteria
- [ ] Metric 1
- [ ] Metric 2

## Related PRD
Link to PRD if exists
```

## Best Practices

1. **Use consistent naming** - [TYPE-NNN] format
2. **Be specific** - Reproducible steps for bugs
3. **Include context** - Why this matters
4. **Link related issues** - Show connections
5. **Update status** - Keep it current
